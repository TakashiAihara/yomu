specs.md に登場する技術用語の解説。

## フィード・フォーマット関連

### Feed（フィード）

Web サイトが公開する「新着情報の一覧」を機械可読な形式で配信する仕組み全体を指す。
RSS・Atom・JSON Feed などはすべて「フィードフォーマット」の一種。
購読者はフィードの URL を登録するだけで、サイトを巡回せずに新着記事を受け取れる。

### Entry（エントリ）

フィードの中に含まれる個々の記事・投稿のこと。フィード 1 件に対して複数の Entry が存在する。
miniflux や本アプリでは Feed と Entry を区別してモデル化している（Feed = 購読先、Entry = 記事）。

### RSS（Really Simple Syndication）

最も普及しているフィードフォーマット。バージョンは 1.0 と 2.0 が主流。

- **RSS 2.0**: XML ベース。ブログや podcast で広く使われる。`<channel>` の下に `<item>` として記事が並ぶ。
- **RSS 1.0 / RDF**: W3C の RDF 仕様に準拠した古いバージョン。現在は少数。

### Atom

RSS とは別の XML ベースのフィードフォーマット（RFC 4287）。
GitHub や YouTube、Blogger など大手サービスで採用されている。
`<feed>` の下に `<entry>` として記事が並ぶ。RSS より仕様が厳密。

### JSON Feed

JSON 形式で書かれたフィードフォーマット（[jsonfeed.org](https://www.jsonfeed.org/)）。
XML を使わずに済むため、実装・デバッグが容易。バージョン 1.0 と 1.1 がある。

### OPML（Outline Processor Markup Language）

フィードの購読リストをまとめてエクスポート・インポートするための XML フォーマット。
「Feedly から乗り換えて全フィードを移す」「バックアップを取る」ときに使う。
RSS リーダー間の標準的なデータ移行手段。

## HTTP 関連

### ETag（Entity Tag）

HTTP レスポンスヘッダの一種。サーバーがコンテンツのバージョンを表すトークン（例: `"abc123"`）を返す。
次回リクエスト時にクライアントが `If-None-Match: "abc123"` を送ると、内容が変わっていなければ
サーバーは `304 Not Modified` を返しボディを省略できる。
フィードポーリングでは「変更がないフィードのデータ転送量をゼロにする」ために使う。

### Last-Modified

HTTP レスポンスヘッダの一種。コンテンツの最終更新日時を示す（例: `Tue, 15 Apr 2025 08:00:00 GMT`）。
次回リクエスト時に `If-Modified-Since: ...` を送ることで ETag と同様の差分取得が可能。
ETag と Last-Modified は**どちらか、または両方**をサポートするサーバーがある。

## データベース関連

### DuckDB

組み込み型の分析指向（OLAP）データベース。サーバープロセスが不要で、単一ファイル（`.duckdb`）に
全データを保存する。SQL 対応・列指向ストレージのため集計クエリが高速。
SQLite に近い手軽さを持ちながら、大量の読み取りや集計に強い。

> 本プロジェクトでは PostgreSQL の代替として採用。DB サーバーなしで動作する点が利点。

### OLAP / OLTP

- **OLTP（Online Transaction Processing）**: 多数の小さな読み書きを高頻度で処理するDB設計。PostgreSQL・MySQL 等。
- **OLAP（Online Analytical Processing）**: 大量データの集計・分析クエリに最適化。DuckDB・BigQuery 等。

DuckDB は OLAP 寄りのため、頻繁な小さい UPDATE（既読化など）を大量に同時実行するのは苦手。
→ 本仕様では **Write Queue（直列化）** で対処する。

### FTS（Full Text Search / 全文検索）

文章の中から単語を検索する仕組み。単純な `LIKE '%keyword%'` と違い、
形態素解析やランキングアルゴリズムを組み合わせて精度の高い検索を実現する。
DuckDB では `PRAGMA create_fts_index(...)` で FTS インデックスを作成できる。

### BM25

FTS でよく使われるランキングアルゴリズム。文書中の単語の出現頻度（TF）と
その単語がどれだけレアか（IDF）を組み合わせてスコアを算出する。
DuckDB の FTS 拡張はデフォルトで BM25 スコアを返す。

### Parquet

列指向のバイナリファイルフォーマット。DuckDB は `COPY table TO 'file.parquet'` で
テーブルを Parquet ファイルにエクスポートできる。pandas・BigQuery 等との連携が容易。

### Write Queue（書き込みキュー）

DuckDB はプロセス間での同時書き込みができないため、書き込みリクエストを一本のキューに集約して
順番に処理する設計パターン。
ポーリングワーカーとユーザー操作の書き込みが衝突しないようにする。

## 認証・セキュリティ関連

### OAuth 2.0

外部サービス（今回は Google）のアカウントを使ってログインする仕組み。
「Google でログイン」ボタンを押すと Google の認証ページに飛び、
許可するとアプリにトークンが渡される。パスワードをアプリ側で管理しなくて済む。

### JWT（JSON Web Token）

認証情報をエンコードした自己完結型のトークン文字列。
サーバーは JWT を発行し、クライアントはリクエストのたびに JWT を送ることで認証する。
セッション情報を DB に毎回問い合わせなくて済むが、失効管理（ログアウト）に工夫が必要。

### API キー

REST API 向けの認証トークン。`Authorization: X-API-Authorization <token>` ヘッダで送る。
CLI やスクリプトから API を叩くときに使う。Google OAuth は Web ブラウザが必要なため、
非対話的な使い方には API キーが適している。

### XSS（Cross-Site Scripting）

悪意のある JavaScript をページに埋め込む攻撃。
フィードの本文には任意の HTML が含まれるため、そのまま表示すると攻撃コードが実行される可能性がある。
→ 保存前に HTML サニタイズ（許可タグ以外を除去）して防ぐ。

## アプリケーション機能関連

### Scraper Rules（スクレイパールール）

`crawler = true` のフィードで全文を取得するとき、元サイトのどの要素を本文として抽出するかを
CSS セレクタで指定するルール。
デフォルトでは `@mozilla/readability` が自動で本文を検出するが、うまくいかないサイトは手動指定する。

### Rewrite Rules（リライトルール）

取得した本文を保存前に変換するルール。
例: 画像 URL を絶対パスに変換する、特定の広告要素を除去する、など。

### Block Filter / Keep Filter（ブロック・キープフィルタ）

- **Block Filter**: 正規表現にマッチしたエントリを**除外**する
- **Keep Filter**: 正規表現にマッチしたエントリ**だけを残す**（それ以外は除外）

フィードレベルとユーザーレベルの 2 段階で設定できる。

### Enclosure（エンクロージャ）

エントリに添付されたメディアファイル（音声・動画・画像）の情報。
Podcast の音声ファイルが代表例で、RSS 2.0 の `<enclosure>` タグ由来の呼称。

### Reading Time（読了時間）

記事本文の文字数から推定する「読み終わるまでの時間」。
平均的な読書速度（日本語: 約 500 文字/分、英語: 約 200 単語/分）を元に算出する。

### Share Code（シェアコード）

エントリをログインなしで外部公開するためのランダムトークン。
`/share/<shareCode>` のような URL でアクセスできるようにする機能。

### Favicon（ファビコン）

ブラウザのタブや bookmarks に表示される小さなアイコン画像（`.ico` / `.png` など）。
フィードアイコンとして購読先サイトの favicon を取得・保存する。

## インフラ・ツール関連

### Hono

Bun・Deno・Cloudflare Workers などで動く軽量 HTTP フレームワーク。
Express に近い API を持ちながら、エッジ環境への対応と高いパフォーマンスを特徴とする。

### tRPC

TypeScript のみで使える型安全な API レイヤー。
REST のように URL + HTTP メソッドでエンドポイントを定義するのではなく、
**ルーター → プロシージャ** の階層構造で API を定義する。

- **Query**: 読み取り専用（REST の GET に相当）
- **Mutation**: 変更あり（REST の POST / PUT / DELETE に相当）

サーバーで定義した入出力の型がクライアントにそのまま伝わるため、
OpenAPI スキーマや API クライアントの手書きが不要。
CLI・Web UI ともに同じ `appRouter` の型を使って呼び出す。

### Drizzle ORM

TypeScript 向けの ORM（Object Relational Mapper）。
SQL に近い記法でクエリを書けるため、生 SQL からの移行が容易。DuckDB ドライバにも対応している。

### Turborepo

monorepo（複数パッケージを1リポジトリで管理する構成）向けのビルドシステム。
パッケージ間の依存関係を解析してタスクを並列実行・キャッシュする。

### Biome

ESLint + Prettier の代替となる Rust 製の Linter / Formatter。
設定ファイルが少なく、処理が高速。

### Polling（ポーリング）

定期的にフィードの URL へ HTTP リクエストを送り、新着エントリを確認する動作。
WebSocket のようなプッシュ型ではなく、クライアント側から能動的に確認しに行く方式。

### Round Robin スケジューラ

全フィードを一定間隔（デフォルト 30 分）で順番に取得するポーリング方式。
シンプルだが、めったに更新されないフィードも頻繁に確認してしまう。

### Entry Frequency スケジューラ

フィードの実際の更新頻度を学習し、次回確認タイミングを動的に調整するポーリング方式。
頻繁に更新されるフィードは短い間隔で、ほとんど更新されないフィードは長い間隔で確認する。
